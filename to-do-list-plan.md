# 双人共享 To-do List 初稿规划

初稿日期：2026-06-29  
项目目标：先做一个仅供两人使用、手机上打开方便、后续可扩展的共享待办工具。

## 1. 核心结论

首选方案：做成 **PWA（Progressive Web App，渐进式 Web 应用）**，而不是一开始就做原生 App。

原因：

- PWA 可以从手机浏览器“添加到主屏幕”，之后像 App 一样从桌面图标进入。
- 不需要先上架 App Store 或 Google Play，因此初期可以避开应用商店审核和开发者账号费用。
- 一套代码可以同时服务 iPhone、Android 和电脑浏览器。
- 对目前“只有我和女朋友两个人共享”的需求来说，PWA 的成本和复杂度最低。
- 未来如果真的需要推送、桌面小组件、系统级提醒、App Store 上架，再考虑封装为原生 App 或跨平台 App。

## 2. 关于“做成 App 是否收费”

截至 2026-06-29 查询到的官方信息：

- Apple Developer Program：如果要正式通过 App Store 等方式分发 App，会员费用为 99 USD/年。来源：https://developer.apple.com/support/compare-memberships/
- Google Play Console：注册开发者账号需要 25 USD 一次性注册费。来源：https://support.google.com/googleplay/android-developer/answer/6112435
- PWA：不通过应用商店也可以使用，只需要部署成 HTTPS 网站，然后在手机上添加到主屏幕。MDN 对 PWA 的描述是：它使用 Web 技术构建，但可以像平台 App 一样安装、离线运行并集成到设备中。来源：https://developer.mozilla.org/en-US/docs/Web/Progressive_web_apps

因此，初期不做原生 App，不走商店上架路线；先做 PWA。

## 3. 产品定位

这是一个“情侣/双人共享待办清单”，重点不是复杂项目管理，而是让两个人快速记录、同步、完成日常事项。

第一版只服务两个人：

- 不开放公开注册。
- 不做多人团队、组织、群组。
- 不做复杂权限系统。
- 只允许两个指定账号进入同一个共享空间。

## 4. 第一版必须完成的功能

### 4.1 账号与访问

- 只允许两个指定邮箱或账号登录。
- 登录后直接进入共享待办列表。
- 未登录用户不能查看任何待办数据。
- 不提供公开注册入口；如需新增用户，必须改配置或后台数据。

### 4.2 待办基础功能

- 新增待办。
- 查看全部待办。
- 标记完成 / 取消完成。
- 删除待办。
- 编辑待办标题。
- 显示创建者。
- 显示创建时间。
- 显示完成者和完成时间。

### 4.3 手机优先体验

- 默认按手机屏幕设计，首屏就能完成“查看 + 新增 + 勾选”。
- 底部或顶部保留快速新增输入区。
- 按钮足够大，适合单手操作。
- 页面不要依赖鼠标悬停。
- 兼容常见手机宽度，例如 360px、390px、430px。
- 可添加到手机主屏幕，并以独立窗口模式打开。

### 4.4 双人共享同步

- 任意一方新增、完成、删除待办，另一方刷新或实时同步后能看到。
- 第一版优先保证数据可靠；实时体验可以先做基础版本，之后再优化。
- 数据必须存在云端数据库，不能只放在某一台手机本地。

## 5. 第一版暂不做的功能

以下功能先不做，避免第一版变重：

- 原生 iOS / Android App。
- App Store / Google Play 上架。
- 公开注册。
- 多人团队。
- 复杂标签系统。
- 日历视图。
- 重复任务。
- AI 自动规划。
- 聊天功能。
- 图片附件。
- 位置提醒。
- 桌面小组件。
- 复杂统计报表。

这些内容可以进入后续版本，但不能阻碍第一版上线。

## 6. 推荐技术路线

### 6.1 前端

- 使用 React + TypeScript + Vite。
- 做成响应式 PWA。
- 使用 `manifest.json` 配置应用名称、图标、启动方式。
- 使用 Service Worker 做基础离线壳缓存。
- UI 风格保持轻、清晰、手机优先，不做复杂视觉装饰。

### 6.2 后端与数据库

优先使用 Supabase：

- 提供登录认证。
- 提供 Postgres 数据库。
- 支持 Row Level Security，适合限制只有两人可访问。
- 支持实时订阅，后续可做即时同步。
- 初期有免费额度，适合小规模双人使用。

备选方案：

- Firebase：实时同步强，但权限规则和数据结构需要认真设计。
- 自建后端：控制力强，但维护成本高，第一版不推荐。

### 6.3 部署

- 前端部署到 Vercel、Netlify 或 Cloudflare Pages。
- 后端数据使用 Supabase 云服务。
- 必须使用 HTTPS，否则 PWA 安装和部分能力会受限。
- 自定义域名可选，不是第一版刚需。

## 7. 数据设计初稿

### 7.1 users

用户由认证服务管理，不单独实现密码系统。

需要记录：

- 用户 ID。
- 昵称。
- 邮箱。
- 头像，可选。

### 7.2 shared_spaces

第一版只有一个共享空间。

字段：

- `id`
- `name`
- `created_at`

### 7.3 space_members

用于限制只有两个人能访问共享空间。

字段：

- `space_id`
- `user_id`
- `role`，第一版可只有 `member`
- `created_at`

### 7.4 tasks

字段：

- `id`
- `space_id`
- `title`
- `notes`，第一版可选
- `status`：`active` 或 `completed`
- `created_by`
- `created_at`
- `updated_at`
- `completed_by`
- `completed_at`
- `deleted_at`，用于软删除，可选

## 8. 页面结构

第一版只需要少量页面：

- 登录页：只负责登录，不做营销页。
- 待办页：主页面，打开后直接看到待办列表。
- 设置页：显示当前登录用户、共享成员、退出登录。

主页面结构：

- 顶部：应用名、当天日期、两人状态或头像。
- 中间：待办列表。
- 底部：快速新增输入框。
- 已完成任务可以折叠，避免干扰当前任务。

## 9. 交互原则

- 新增待办必须足够快：输入文字，按回车或点加号即可。
- 勾选完成必须即时反馈。
- 删除操作需要确认或可撤销，避免误删。
- 网络失败时要提示，不要静默丢数据。
- 两人同时操作同一条任务时，以数据库最新更新时间为准。
- 第一版不追求花哨动画，优先稳定、清楚、顺手。

## 10. 隐私与安全原则

- 不公开任何待办内容。
- 所有数据读取和写入必须验证登录用户是否属于共享空间。
- 数据库必须启用行级安全策略。
- 前端不能依赖“隐藏按钮”作为权限控制。
- 不在代码中写死密钥。
- 不记录过度隐私数据。

## 11. 开发阶段安排

### 阶段 0：确认基础决策

目标：

- 确定使用 PWA。
- 确定两人使用的登录邮箱。
- 确定应用名称。
- 确定是否使用 Supabase。

完成标准：

- 本规划文件被确认。
- 若有修改，先更新本文件。

### 阶段 1：项目初始化

目标：

- 创建前端项目。
- 配置 TypeScript、基础样式、路由。
- 配置 PWA manifest。
- 做出手机优先的空状态页面。

完成标准：

- 本地可以打开页面。
- 手机尺寸预览无明显布局问题。

### 阶段 2：数据库与登录

目标：

- 创建 Supabase 项目。
- 配置登录方式。
- 建立数据表。
- 配置 Row Level Security。
- 限制只有两个账号能访问共享空间。

完成标准：

- 未登录无法访问待办。
- 非成员无法读取或写入数据。

### 阶段 3：待办 MVP

目标：

- 新增待办。
- 查看待办。
- 完成待办。
- 删除待办。
- 编辑待办。
- 显示创建者和完成者。

完成标准：

- 两个账号都可以正常使用同一个列表。
- 基础异常场景有提示。

### 阶段 4：PWA 与手机体验

目标：

- 添加应用图标。
- 添加到主屏幕后可正常打开。
- 优化手机输入和触控体验。
- 增加基础离线壳缓存。

完成标准：

- iPhone / Android 至少一种真机或模拟视图验证可用。
- 从主屏幕打开时体验接近 App。

### 阶段 5：上线与验收

目标：

- 部署到线上 HTTPS 地址。
- 两人实际使用测试。
- 修复第一轮体验问题。

完成标准：

- 两人均能从手机主屏幕打开。
- 两人均能新增、完成、删除待办。
- 没有第三人能访问数据。

## 12. 后续版本候选功能

后续可以按真实使用中的痛点选择，不提前堆功能：

- 到期时间。
- 任务分类。
- 重要程度。
- 重复任务。
- 任务备注。
- 任务搜索。
- 完成记录。
- 每日/每周回顾。
- 推送提醒。
- 情侣纪念日或共同计划模块。
- 桌面小组件。
- 原生 App 封装。

## 13. 未来行动规则

后续开发必须遵守以下规则：

- 所有实现优先服务“手机上两个人顺手使用”。
- 第一版不得为了未来大功能牺牲当前简单性。
- 如果要改变技术路线，先修改本文件，再行动。
- 如果要新增功能，先判断是否属于第一版；不属于则进入“后续版本候选功能”。
- 如果实现过程中发现本规划不合理，先更新规划，再改代码。
- 安全和隐私优先级高于界面效果。
- 数据可靠性优先级高于动画、主题和装饰。

## 14. 当前推荐的下一步

下一步建议先做阶段 0：

- 确定应用名称。
- 确定两人的登录邮箱。
- 确认是否使用 Supabase。
